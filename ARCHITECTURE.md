# Архитектура MyMind

MyMind — локальное Electron-приложение. Renderer построен на React, main process владеет SQLite и файловым хранилищем, а preload публикует узкий типизированный IPC API.

## Границы процессов

- `src/main` — создание окна, миграции, репозитории, файловые операции и политики безопасности.
- `src/preload` — единственный мост renderer → main. `contextIsolation`, `sandbox` включены, Node.js в renderer выключен.
- `src/renderer` — интерфейс и локальное состояние редактирования. Модули описаны в `app/module-registry.ts` и загружаются лениво.
- `src/shared` — контракты IPC, Zod-схемы и безопасные платформонезависимые утилиты.

## Данные и сохранение

Материалы хранятся в SQLite как версионированный документ. Renderer последовательно ставит autosave в очередь; main дополнительно сериализует сохранение, удаление и очистку ресурсов по material ID. После транзакции очистка получает именно сохранённый документ, поэтому более старый запрос не может удалить ресурс новой версии.

SQLite работает в WAL, с foreign keys и `busy_timeout`. Миграции Drizzle запускаются до регистрации рабочего окна. Изменение схемы требует `npm run db:generate`, проверки с существующей копией базы и `npm run db:check`.

## Локальные ресурсы

SQLite хранится в Electron `userData/data/mymind.sqlite`, а пользовательские вложения — отдельно в системном каталоге документов: `Documents/MyMind/Attachments`. Вложения выдаются через `mymind-asset://`: URL канонизируется, имена и ID валидируются, путь обязан остаться внутри каталога конкретного материала. Сохранение проверяет существование файла и совпадение владельца. Протокол поддерживает только GET/HEAD и byte ranges.

## Расширение

Новый верхнеуровневый модуль добавляется одной записью в `app/module-registry.ts`: из ключей реестра выводится `AppViewId`, а навигация и lazy-rendering используют те же определения. Новый тип study-блока требует shared-контракт и Zod-схему, а exhaustive-реестры TypeScript требуют фабрику, read/edit/settings-стратегии и main/domain extractor plain text; отсутствие записи становится ошибкой компиляции.

## Autosave, навигация и завершение работы

Каждый `StudyMaterialEditor` владеет отдельной `StudyAutosaveQueue`, замкнутой на неизменяемый `materialId`. Debounce только добавляет версии черновика; `flushLatestDraft()` останавливает таймер и последовательно сохраняет последнюю версию. Поэтому документ материала A не может быть отправлен с ID материала B.

Смена материала, переход к папке/главной, создание, дублирование и смена верхнеуровневого модуля проходят через transition coordinator. Dirty-редактор остаётся смонтированным, пока flush не завершён. При ошибке переход отменяется, черновик остаётся в памяти, а UI предлагает повторить сохранение, остаться или после повторного явного подтверждения покинуть страницу с потерей изменений.

Закрытие окна использует IPC handshake. Main перехватывает `close`, `before-quit` и, насколько позволяет Electron, `query-session-end`, отправляет renderer запрос flush и удерживает окно открытым. Renderer отвечает `success`, `failed`, `cancel` или явно выбранным `force`. SQLite закрывается только перед окончательным закрытием после `success`/`force`; ответ принимается только от main frame текущего доверенного `WebContents` и валидируется Zod.

## Жизненный цикл импортированных вложений

Picker работает вне material coordinator, а проверка существования материала и копирование — внутри него. После успешного копирования main создаёт reservation `materialId + assetId + createdAt` на 10 минут. Очистка постоянного каталога учитывает и ссылки сохранённого документа, и активные reservations, поэтому stale autosave без нового блока не удаляет только что импортированный файл. Reservation снимается после успешного сохранения документа со ссылкой, очищается при удалении материала или истекает по TTL; orphan удаляется следующей безопасной очисткой. Дублирование копирует только вложения из сохранённого документа и не видит pending reservations.

## Обслуживание legacy plain text

После миграций и до показа UI запускается versioned maintenance `study_plain_text_maintenance_v1`. Она выбирает только legacy-записи с `plain_text` больше актуального лимита, валидирует исходный документ и пересчитывает индекс через `documentToPlainText`, не меняя JSON документа. Повреждённые документы сохраняются без изменений и логируются только по ID и типу ошибки. Маркер в `app_meta` делает проход идемпотентным.
