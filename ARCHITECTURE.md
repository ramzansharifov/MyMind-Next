# Архитектура MyMind

MyMind — локальное Electron-приложение. Renderer построен на React, main process владеет SQLite и файловым хранилищем, а preload публикует узкий типизированный IPC API.

## Границы процессов

- `src/main` — создание окна, миграции, репозитории, файловые операции и политики безопасности.
- `src/preload` — единственный мост renderer → main.
- `src/renderer` — интерфейс и локальное состояние редактирования.
- `src/shared` — IPC-контракты, Zod-схемы и платформонезависимые утилиты.

Electron BrowserWindow использует:

- `contextIsolation: true`;
- `nodeIntegration: false`;
- `sandbox: true`.

Renderer не получает прямой доступ к Node.js или `ipcRenderer`.

## IPC

Preload публикует типизированный `window.api`.

Main process:

1. принимает payload;
2. проверяет доверенный sender там, где это необходимо;
3. валидирует данные через Zod;
4. вызывает repository или service;
5. возвращает сериализуемый результат.

Обычные IPC handlers выполняются через `MainOperationTracker`.

Shutdown response IPC не проходит через tracker, иначе renderer не смог бы подтвердить завершение после блокировки новых операций.

## Данные и сохранение

Материалы хранятся в SQLite как версионированные документы.

Renderer последовательно ставит autosave в очередь. Main process дополнительно сериализует сохранение, импорт, удаление и дублирование по material ID.

После транзакции cleanup получает фактически сохранённый документ. Старый save не может удалить файл, используемый новой подтверждённой версией документа.

SQLite использует:

- WAL;
- foreign keys;
- `busy_timeout`;
- синхронные транзакции `better-sqlite3`.

Миграции Drizzle запускаются до показа рабочего окна.

## MainOperationTracker

`MainOperationTracker`:

- считает активные IPC-операции;
- позволяет дождаться состояния idle;
- запрещает новые операции во время финальной стадии shutdown;
- снова разрешает операции, если пользователь отменяет закрытие.

Операция считается активной до завершения возвращённого Promise, включая файловое копирование и cleanup.

## Autosave и навигация

Каждый `StudyMaterialEditor` создаёт отдельную `StudyAutosaveQueue`, привязанную к неизменяемому `materialId`.

Очередь различает:

- последнюю версию черновика;
- последнюю поставленную в сохранение версию;
- последнюю успешно сохранённую версию;
- активный save;
- mounted-состояние редактора;
- deletion pause;
- окончательный dispose.

Mounted-состояние и deletion pause независимы.

`deactivate()` означает только unmount редактора и никогда не включает deletion pause.

`resume()` снимает только deletion pause и никогда не выполняет повторный mount очереди.

Смена материала или верхнеуровневого модуля сначала выполняет flush активного dirty draft.

Tracked draft handle использует deletion epoch и хранит последний deletion barrier даже после его завершения.

Это закрывает обе последовательности:

```text
delete начинается → transition flush начинается
```

и:

```text
transition flush начинается → delete начинается во время сохранения
```

После каждого асинхронного `source.flush()` handle проверяет, не изменился ли deletion epoch.

Если удаление завершилось commit:

- переход продолжается;
- дополнительный save удалённого материала не запускается.

Если удаление завершилось rollback:

- handle повторяет source flush;
- восстановленный материал гарантированно сохраняет последнюю версию;
- повторный flush безопасно становится no-op, когда rollback уже сохранил документ.

Ошибка rollback передаётся ожидающему переходу или shutdown.

## Удаление активного материала

Удаление активного материала отличается от обычного перехода: пользователь явно подтвердил удаление, поэтому dirty draft не нужно сохранять перед удалением.

Перед IPC delete renderer создаёт deletion suspension:

1. отменяет debounce timer;
2. переводит autosave queue в deletion pause;
3. сохраняет последний документ в памяти;
4. запрещает новые save-запросы;
5. отправляет delete в main process.

Если удаляется папка, renderer проверяет всё поддерево. Suspension создаётся, когда active material ID находится внутри удаляемой папки.

Зарегистрированный `StudyDraftHandle` хранит Promise результата удаления.

Пока delete не завершён, handle сообщает о наличии незавершённой работы, даже когда документ был чистым.

Следующие операции ждут результата delete:

- переход к другому материалу;
- переход в другой верхнеуровневый модуль;
- shutdown flush.

### UI delete coordinator

`StudyPage` держит confirmation dialog открытым до ответа backend.

Сразу после подтверждения:

- синхронный ref закрывает окно повторного двойного клика;
- устанавливается `isDeleting`;
- кнопки подтверждения и отмены блокируются;
- Shift+Enter перестаёт запускать повторное удаление;
- Escape и pointer outside не закрывают диалог;
- модальный overlay блокирует дерево и workspace;
- программные study transitions дополнительно проверяют delete ref.

Диалог закрывается только после подтверждённого успешного backend delete.

При ошибке:

- диалог остаётся открытым;
- отображается сообщение;
- suspension выполняет rollback;
- материал и его черновик остаются доступными;
- пользователь может повторить удаление или отменить диалог.

`AlertDialog.Action` намеренно не используется для async delete, потому что он закрывает диалог синхронно до завершения backend-запроса.

### Успешное удаление

При успешном ответе backend:

- suspension выполняет `commit`;
- queue окончательно `dispose`;
- ожидающие flush завершаются без сохранения удалённого документа;
- узел и потомки удаляются из renderer state;
- selected node очищается, если находился в удалённом поддереве;
- диалог закрывается;
- редактор размонтируется без запоздалого autosave.

### Ошибка удаления

При ошибке backend:

- suspension выполняет `rollback`;
- deletion pause снимается;
- если редактор всё ещё смонтирован, последняя версия сохраняется;
- ожидающие переходы продолжаются только после rollback;
- диалог остаётся открытым с ошибкой.

Если редактор уже размонтирован, rollback не может повторно активировать очередь.

Активный IPC save нельзя физически отменить. Main-process material coordinator гарантирует последовательное выполнение save и delete одного material ID.

## React Strict Mode

Renderer запускается внутри `React.StrictMode`.

Strict Mode в development повторно вызывает effect setup и cleanup. Поэтому обычный cleanup редактора использует обратимую `deactivate()`, а не окончательный `dispose()`.

`activate()` не снимает настоящий deletion pause.

Окончательный dispose выполняется только после подтверждённого удаления материала.

## Завершение приложения

Main process перехватывает:

- закрытие BrowserWindow;
- `before-quit`;
- `query-session-end`.

Последовательность обычного завершения:

1. main создаёт UUID shutdown request;
2. request отправляется renderer;
3. renderer выполняет flush активного draft;
4. pending delete сначала завершается commit или rollback;
5. renderer отвечает `success`, `failed`, `cancel` или `force`;
6. после `success` main запрещает новые обычные IPC-операции;
7. main ожидает завершения уже начатых операций;
8. SQLite закрывается;
9. окно уничтожается;
10. приложение завершается.

Ответ shutdown принимается только:

- от текущего доверенного `WebContents`;
- из main frame;
- после Zod-валидации request ID и decision.

Ожидание renderer и ожидание операций имеют отдельные timeout.

Native fallback используется, когда:

- renderer не ответил;
- renderer стал `unresponsive`;
- renderer process завершился;
- операции выполняются дольше допустимого времени.

Пользователь может:

- подождать;
- отменить закрытие;
- закрыть приложение принудительно.

При выборе «Подождать ещё» main повторно отправляет тот же shutdown request renderer и запускает новый timeout.

При отмене tracker снова принимает новые операции.

## Локальные ресурсы

SQLite хранится в:

```text
Electron userData/data/mymind.sqlite
```

Пользовательские вложения находятся в:

```text
Documents/MyMind/Attachments
```

Вложения выдаются через `mymind-asset://`.

URL канонизируется, ID и имена файлов валидируются, а разрешённый путь обязан оставаться внутри каталога соответствующего материала.

Протокол поддерживает:

- `GET`;
- `HEAD`;
- byte ranges.

После импорта main создаёт временную reservation:

```text
materialId + assetId + createdAt
```

Cleanup сохраняет:

- assets, используемые подтверждённым документом;
- assets с активной reservation.

Поэтому stale autosave не может удалить только что импортированный файл.

Reservation:

- снимается после успешного save со ссылкой на asset;
- удаляется вместе с материалом;
- истекает по TTL;
- не копируется при дублировании материала.

## Legacy plain text maintenance

После миграций выполняется versioned maintenance производного поля `plain_text`.

Maintenance:

- выбирает oversized legacy-значения;
- валидирует исходный документ;
- пересчитывает plain text;
- не меняет JSON документа;
- не логирует пользовательское содержимое;
- сохраняет версию выполнения в `app_meta`.

## Расширение приложения

Новый верхнеуровневый модуль добавляется в `app/module-registry.ts`.

Из ключей registry выводится `AppViewId`, поэтому навигация и lazy rendering используют один источник истины.

Новый study block требует:

- shared-контракт;
- Zod-схему;
- renderer registry entry;
- factory;
- edit strategy;
- read strategy;
- settings strategy;
- main plain-text extractor.

Exhaustive TypeScript mappings превращают отсутствие обязательной реализации в ошибку компиляции.
